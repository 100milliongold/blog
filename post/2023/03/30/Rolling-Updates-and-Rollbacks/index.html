Rolling Updates

2023년 3월 30일 · 약 6분
100milliongold
developer

-

배포의 중요한 측면은 업그레이드, 롤아웃, 롤백입니다.

업그레이드 : 최신 주 버전을 배포하고 이전 버전을 종료하는 기능입니다.

롤아웃 : 다운타임 없이 최신 마이너 버전(버그 수정, 핫픽스, 마이너 기능, 향상)을 배포하는 기능.

롤백 : 문제가 발생할 경우 이전 작업 버전을 다시 복원하는 기능입니다.

다음은 K8s 의 세계에서 일어나는 일입니다.**

  1. 배포를 처음 생성하면 롤아웃이 생성됩니다.
  2. 새 롤아웃은 새 버전을 생성합니다.
  3. 향후 동일한 이름의 새 배포가 트리거되면 버전이 증가된 새 롤아웃이 생성됩니다.
  4. 이를 통해 변경 사항을 추적하고 이전 버전 배포로 롤백할 수 있습니다.
  5. 롤아웃 상태 확인: kubectl rollout status deployment <deployment-name>
  6. 롤아웃 내역, 개정 및 변경 원인 확인: kubectl rollout history deployment <deployment name>

배포 전략 #1 -> Recreate:

  1. 실행 중인 앱 인스턴스가 5개 있다고 가정합니다.
  2. 새 버전을 배포할 때 이전 버전의 인스턴스 5개를 제거한 다음 새 버전의 인스턴스 5개를 배포할 수 있습니다.
  3. 문제는 중단 시간이 있다는 것입니다.
  4. 이는 주로 주요 변경, 주요 변경 또는 이전 버전과의 호환성이 불가능할 때 수행됩니다.
  5. 이것은 K8의 기본 전략이 아닙니다.

배포 전략 #2 -> Rolling update:

  1. 이 전략에서는 이미 실행 중인 모든 인스턴스를 삭제하지 않습니다.
  2. 한 번에 특정 비율로 인스턴스를 삭제하고 동일한 비율의 최신 버전 포드를 동시에 생성합니다.
  3. 이 업그레이드는 K8s의 기본 전략입니다.
  4. 가동 중지 시간이 없습니다.

예를 들어 이해해 보겠습니다.

  1. 이미지가 nginx:1.7.0인 포드의 복제본 3개를 실행하는 기존 배포가 이미 있다고 가정합니다.

  2. 이제 이미지의 버전을 변경하려고 합니다.

  3. 배포 파일에서 이미지 버전을 변경하고 다음 명령을 실행하면 됩니다.: kubectl apply -f <배포 파일 경로>

  4. 위의 작업은 다음 명령어로도 수행할 수 있습니다.: kubectl set image deployment myapp-deployment nginx=nginx.1.7.1.

  5. 4단계를 수행하면 실제 파일과 클러스터의 배포 정의에 불일치가 있음을 기억하십시오.

  6. 다음 명령을 실행하여 배포 세부 정보를 확인하고 두 전략의 차이점을 확인합니다. : kubectl describe deployment <deployment name>

업그레이드 작동 방식:

  1. 배포가 적용되면 복제 세트를 생성하고 배포 구성에 언급된 인스턴스 수로 포드를 가동합니다.

  2. 그런 다음 배포가 변경 사항과 함께 다시 적용되면 배포 구성에 언급된 대로 새 복제본 세트를 생성하고 인스턴스 수로 포드를 스핀업하고 동시에 이전 복제본 세트에서 포드를 삭제합니다.

  3. 그러나 주목해야 할 점은 필요한 경우 롤백에 사용되는 이전 복제본 세트가 여전히 존재한다는 것입니다.

  4. 배포를 롤백하려면: kubeclt rollout undo deployment <deployment name> — 업그레이드 중에 발생한 것과 유사한 순서로 실행됩니다.

  5. 롤백 후에도 새 복제 세트는 계속 유지됩니다.

  6. 기록 개정의 변경 원인을 보려면 배포를 편집/적용하는 동안 레코드 플래그를 추가해야 합니다(배포당 한 번 설정해야 함).

  7. 롤백을 수행하면 롤백이 발생한 리비전이 기록에서 제거되고 대신 기록에 새 항목이 만들어집니다.

  8. 업그레이드 중에 오류가 발생하면 kubernetes는 사전에 업그레이드를 중지하고 이전에 실행 중인 인스턴스 삭제를 중지합니다.
